# Bug Sweep

对整个仓库执行系统性的 bug 巡查，通过并行代码审查发现已确认的缺陷和高置信度风险。

## 目标

查找以下问题：

- 功能性 bug
- 行为回归
- 契约不匹配（接口、API、类型约定）
- 缺失的输入验证
- 错误的错误处理
- 不安全的假设（竞态、空指针、未检查的返回值等）
- 缺失或薄弱的测试覆盖（可能隐藏缺陷）

## 非目标

- 不是测试执行工作流——不负责运行和报告测试结果
- 不是文档重写工作流——不负责同步文档
- 不要把 bug 发现埋在宽泛的文档编辑中
- 不要仅审查最近变更的文件——必须覆盖整个仓库

## 核心规则

### 信息源

- 代码是唯一权威信息源
- 已有文档可能过时，必须对照代码验证
- 已有测试可能不完整，不能当作正确性证明

### 审查范围

- "全量" 意味着影响交付行为的整个仓库
- 必须覆盖入口点（CLI/API/HTTP）、核心逻辑、安全边界、持久化层、错误路径
- 除非用户明确缩小范围，否则不限制为最近变更的文件

### 委派

- 主智能体保留判断密集型决策、综合分析和最终结论
- 子智能体用于机械性代码扫描和局部审查，仅在用户选择委派执行时使用
- 代码审查任务应使用专用子智能体并行扫描不同代码区域

### 输出纪律

- 发现必须区分 **已确认问题** 和 **假设**
- 所有发现必须有代码证据支撑，不推测
- 按严重程度排序：Critical > High > Medium > Low

## 工作流

### 步骤 1：识别代码区域

读取仓库结构，将代码划分为可并行审查的区域。典型划分：

1. **入口层** — HTTP 处理器、CLI 命令、请求验证、错误响应
2. **核心逻辑** — 业务规则、状态机、工作流引擎、数据处理
3. **安全与策略** — 权限校验、输入清洗、访问控制、敏感数据处理
4. **持久化层** — 数据库操作、状态存取、迁移、事务完整性
5. **边界与集成** — 外部 API 调用、并发控制、超时处理、资源清理

根据实际仓库结构调整区域划分。区域之间应尽量独立，以支持并行扫描。

### 步骤 2：并行扫描

每个区域启动一个子智能体（或在主智能体中顺序审查）：

- 扫描该区域的所有相关代码
- 仅报告 **已确认的 bug** 和 **高置信度风险**
- 每条发现必须附带：位置（文件:行号）、现象描述、严重程度、代码证据

### 步骤 3：汇总与分级

主智能体汇总所有子智能体的发现：

1. 去重和关联（同一根因的不同表现应合并）
2. 按严重程度分级
3. 区分已确认问题与假设性风险
4. 识别测试覆盖缺口

### 步骤 4：输出报告

结构化报告包含：

- **已确认 Bug** — 按严重程度排序，每条含位置、描述、证据
- **高置信度风险** — 可能但未完全确认的问题
- **测试覆盖缺口** — 缺少测试保护的关键路径
- **文档漂移** — 代码与文档不一致之处（如有）

## 完成条件

- 所有代码区域已扫描
- 所有发现有证据支撑并按严重程度排序
- 已确认问题与假设性风险已明确区分

## 常见错误

- 仅扫描最近变更的文件而忽略其他代码
- 将推测当作确认发现报告
- 在发现中夹杂文档重写或测试执行（应聚焦 bug 发现）
- 未能识别跨区域的同一根因问题
- 跳过错误路径和边界条件的审查
