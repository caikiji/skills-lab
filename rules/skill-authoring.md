# 技能编写规范

## 简洁原则

- 上下文窗口是共享资源，技能与系统提示、对话历史、其他技能竞争上下文。
- 只添加 Claude 不知道的上下文。质疑每一条信息："Claude 真的需要这个解释吗？"
- SKILL.md 正文控制在 500 行以内。超过时拆分为单独的参考文件。

## 描述编写

- `description` 字段**必须用第三人称编写**。描述注入系统提示，视角不一致会影响发现。
- 描述必须具体并包含关键术语，说明技能的功能**和**使用时机。
- 避免模糊描述如"帮助处理文档"。

## 自由度设置

将具体程度与任务的脆弱性匹配：

- **高自由度**：多种方法都有效、决策依赖上下文时，用基于文本的说明。
- **中自由度**：存在首选模式、某些变化可接受时，用伪代码或带参数的脚本。
- **低自由度**：操作脆弱易错、一致性至关重要时，提供精确的脚本和命令。

## 渐进式披露

- SKILL.md 作为概述，指向 Claude 按需查看的详细材料。
- 所有参考文件**保持一级深度**——直接从 SKILL.md 链接，不要嵌套引用。
- 超过 100 行的参考文件，在顶部包含目录。

## 命名约定

- 技能名称使用**动名词形式**（动词 + -ing），如 `processing-pdfs`、`analyzing-spreadsheets`。
- 名称只能包含小写字母、数字和连字符，最多 64 个字符。
- 避免模糊名称（`helper`、`utils`）和保留字（`anthropic`、`claude`）。

## 反模式

- **避免时间敏感信息**：不要包含会过时的内容。已过时的信息放在"旧模式"部分。
- **避免提供太多选项**：提供默认方案而非多个替代方案。
- **避免 Windows 风格路径**：始终使用正斜杠 `scripts/helper.py`，不用反斜杠。
- **使用一致的术语**：选定一个术语后全文统一，不要混用同义词。
- **不要假设工具已安装**：明确列出依赖项和安装方式。
