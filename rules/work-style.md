# 工作规范

- 执行过程中创建的临时文件、脚本或辅助文件，**必须**在任务结束时删除清理。

## 代码声明必须基于事实

- 绝不要对没有读取过的代码进行推测。
- 用户引用特定文件时，**必须**先读取该文件再回答。
- 回答有关代码库的问题前，**必须**调查和读取相关文件。
- 除非确定正确答案，否则在调查前不要对代码做任何声明——给出有根据的、无幻觉的答案。

## 实现质量

- 使用可用的标准工具编写高质量的通用解决方案，不要创建辅助脚本或变通方法来走捷径。
- 实现对所有有效输入都能正确工作的方案，而非仅适配测试用例。不要硬编码值或创建仅适用于特定测试输入的方案。
- 专注于理解问题需求并实现正确的算法。测试用于验证正确性，不是用来定义解决方案。
- 遵循最佳实践和软件设计原则，提供健壮、可维护、可扩展的实现。
- 如果任务不合理或不可行，或任何测试不正确，应告知用户而非绕过。

## 持续性与自主性

- 上下文窗口接近限制时会自动压缩，因此**不要**因 token 预算问题提前停止任务。
- 接近 token 预算限制时，将当前进度和状态保存到记忆中，以便上下文刷新后继续。
- 始终尽可能保持持久性和自主性，完整地完成任务。
- 无论剩余上下文如何，**不得**人为地提前停止任何任务。

## 长任务执行

- 对于长任务，先清晰规划再执行。鼓励花费整个输出上下文来处理任务。
- 系统地持续推进，直到任务完成。不要在有大量未提交工作的情况下耗尽上下文——适时提交阶段性成果。
- 将长任务分解为可独立提交的步骤，每完成一步即提交，避免因上下文压缩丢失进度。
